MODULE 4 BLUEPRINT - OVERRIDES (GAME CONFIG CONTROL PLANE)
===========================================================
Phase 3 scope (with Module 5). The "edit live from the dashboard"
feature. Core design invariant from the plan: VERSIONS ARE IMMUTABLE.
Every save is a new version; activation moves a pointer; rollback is
repointing at an old version. Nothing is ever edited in place - that
invariant is what makes "rollback restores prior values exactly" a
one-line proof instead of a prayer.

----------------------------------------------------------------------
A. UNITS - package dev.shayveri.core.overrides
----------------------------------------------------------------------

O1. ConfigSaveRequest               (record)
    PUT /api/config body: placeId (nullable -> global), namespace
    (@NotBlank: weapons|spawns|graphics|...), values (@NotNull Map).
    Consumes: validation + Jackson.

O2. ConfigVersion                   (Mongo document, IMMUTABLE)
    Collection: config_versions. Fields: @Id id; String placeId;
    String namespace; int version (monotonic per placeId+namespace);
    Map values; String savedBy (api key label - the audit trail
    seed); Instant savedAt. NO setters at all - constructor only; the
    class itself enforces the invariant.
    Consumes: @Document, @Id, @Indexed on (placeId, namespace).

O3. ConfigSchemaRegistry            (ours - the typo firewall)
    validate(namespace, values) -> list of problems. v1 is deliberately
    simple: per-namespace map of known keys -> expected primitive type,
    registered in code. Unknown key or wrong type -> reject BEFORE a
    version is created; a typo'd key can never reach a live game (plan's
    explicit requirement). Full JSON Schema is a v2 swap behind this
    same unit.
    Consumes: nothing - ours.

O4. ConfigStore                     (interface - rule 5)
    saveVersion(ConfigVersion), findVersion(placeId, ns, version),
    latestVersionNumber(placeId, ns), history(placeId),
    setActivePointer(placeId, ns, version) + getActivePointer(...)
    (pointer is a tiny separate Mongo doc - durable truth; Redis is
    only the cache of it).
O5. MongoConfigStore (+ repositories) - established pattern.

O6. ActiveConfigCache               (interface - rule 5, hot path)
    put(placeId, assembledConfig, etag), get(placeId) -> {body, etag}.
O7. RedisActiveConfigCache
    Key config:active:{placeId} holding the ASSEMBLED config (all
    namespaces' active versions merged) + its etag, refreshed on every
    activation. No TTL - explicitly invalidated/replaced, never expired
    (an expired cache would silently fall to Mongo on the hottest
    ROBLOX path).
    Consumes: StringRedisTemplate.opsForValue(); Jackson ObjectMapper
    (writeValueAsString for the cached body - one of the rare direct
    Jackson uses).

O8. ConfigService
    save(req, who): O3.validate -> version = latest+1 -> new immutable
        O2 -> O4.saveVersion. Returns the version number. NOTHING is
        activated by saving - save and activate are separate acts.
    activate(placeId, ns, version, who): O4.setActivePointer ->
        reassemble + O6.put (new etag = hash of body) -> Module 5's
        EgressService.publishConfigActivated(placeId, version) ->
        /topic/config -> Module 7 audit hook (before/after = old and
        new version numbers).
    rollback IS activate(old version) - no separate code path to test.
    getActive(placeId) -> from O6, fall through to O4 + repopulate on
        cache miss.
    Consumes: @Service.

O9. ConfigController                (HARD-ISH SPOT: the ETag flow)
    GET /api/config/active?placeId= (ROBLOX) - the poll path that must
        be ~free when nothing changed:
        request carries If-None-Match: "<etag it last saw>";
        if it equals the cached etag -> 304 Not Modified, EMPTY body
        (bytes on the wire: ~none, Mongo untouched, this is the whole
        point); else 200 + body + ETag header.
        Consumes: ResponseEntity.status(HttpStatus.NOT_MODIFIED),
        .eTag(v), @RequestHeader(HttpHeaders.IF_NONE_MATCH, required =
        false) - Spring Web has first-class ETag pieces, no manual
        header parsing.
    PUT /api/config (DASH), POST /api/config/activate/{version} (DASH),
    GET /api/config/history (DASH).
    SecurityConfig: active -> ROBLOX; the rest -> DASH.

----------------------------------------------------------------------
B. TEST PLAN
----------------------------------------------------------------------
T1. Save twice -> versions 1,2 both retrievable, byte-identical to what
    was saved (immutability).
T2. Schema: unknown key "zombeSpeed" -> 400 naming it; no version row
    created.
T3. Activate v1 -> GET active serves v1 within one poll; activate v2
    then rollback to v1 -> body EXACTLY equals T1's v1 (acceptance).
T4. ETag: GET -> 200 + etag; GET with If-None-Match=etag -> 304 empty;
    after activation -> 200 again with new etag.
T5. Activation publishes /topic/config and calls egress (egress mocked
    at ITS interface - Module 5 has its own tests; here only "was
    called with placeId+version" matters).
T6. Roles: ROBLOX cannot PUT; DASH cannot GET active? (per plan active
    is ROBLOX - decide DASH read access at implementation, default
    deny).
