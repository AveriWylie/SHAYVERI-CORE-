MODULE 6 BLUEPRINT - REALTIME, FULL BUILD (DASHBOARD WEBSOCKET HUB)
====================================================================
Phase 1 already delivered the minimal slice: RealtimePublisher (B1),
StompRealtimePublisher (B2), WebSocketConfig (B3), open handshake. This
blueprint is the REMAINDER that makes Module 6 complete, built whenever
the dashboard work demands it (naturally alongside Phase 2-3). Topic

----------------------------------------------------------------------
A. UNITS - package dev.shayveri.core.realtime (+ one config touch)
----------------------------------------------------------------------

R1. StompAuthInterceptor            (closes the Phase-1 security hole)
    The seam B3 left obvious: CONNECT frames currently pass unchecked.
    Design: intercept every inbound STOMP frame; on CONNECT, read the
    native header X-Api-Key, resolve it (the ApiKeyResolver seam from
    the security architecture - resolver, not raw map), require DASH,
    else reject the connection. Non-CONNECT frames pass through
    (session is already vetted).
    Consumes: ChannelInterceptor (override preSend);
    StompHeaderAccessor.wrap(message) - .getCommand() == StompCommand
    .CONNECT, .getFirstNativeHeader("X-Api-Key"); register it in B3's
    class via configureClientInboundChannel(ChannelRegistration
    registration) -> registration.interceptors(this). Rejection =
    return null / throw - the frame dies before the broker sees it.
    NOTE: HTTP security (SecurityConfig) does NOT cover STOMP frames -
    the /ws handshake is HTTP, but frames ride inside the socket after
    upgrade. This interceptor is the STOMP-side equivalent of
    ApiKeyAuthFilter, and the same discipline applies: auth lives here,
    never in message handlers.

R2. SnapshotController              (the reconnect-hydration path)
    GET /api/snapshot (DASH): one JSON of CURRENT full state so the
    dashboard hydrates before deltas stream in:
      { nodes: [Module 2 listWithStatus()],
        queue: {depths per type, active job summaries} (Module 3),
        config: {active versions per place} (Module 4),
        alerts: recent N }
    Assembled EXCLUSIVELY through the other modules' interfaces
    (NodeService, JobStore, ConfigStore) - realtime owns no data, it
    aggregates views. Recent-alerts needs a tiny ring buffer:
    R3. AlertBuffer (ours): last 50 alerts in memory, fed by the same
    publish path; loss on restart acceptable (alerts are ephemeral by
    nature - durable history is Module 7's audit).
    Consumes: Spring Web set; SecurityConfig line -> DASH.

R4. Broker seam note (no code yet): the plan's scale escape hatch is
    one line in B3 - enableSimpleBroker -> enableStompBrokerRelay
    (external Redis/RabbitMQ-backed broker) if the dashboard ever
    outgrows in-memory. Documented so nobody designs around the
    in-memory broker's limits by accident.
