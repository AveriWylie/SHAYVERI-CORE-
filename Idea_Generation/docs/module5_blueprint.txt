MODULE 5 BLUEPRINT - EGRESS (ROBLOX OPEN CLOUD CLIENT)
=======================================================
Phase 3 scope (with Module 4). Smallest module: internal-only (no
controller, no endpoints), one outbound call, wrapped in the reliability
pieces. Design law from the plan: PUSH FAILURE IS NEVER FATAL - the
Module 4 poll path is the guaranteed fallback, so everything here
degrades loudly instead of breaking anything.

----------------------------------------------------------------------
A. UNITS - package dev.shayveri.core.egress
----------------------------------------------------------------------

E1. OpenCloudProperties             (config binding)
    shayveri.opencloud.{api-key, universe-id, topic} from
    application.yml (already scaffolded there; values from env, never
    committed).
    Consumes: @ConfigurationProperties(prefix = "shayveri.opencloud")
    + @EnableConfigurationProperties - the ApiKeyProperties pattern
    verbatim.

E2. OpenCloudClient                 (the HTTP call)
    publish(placeId, version): POST
    https://apis.roblox.com/messaging-service/v1/universes/{universeId}/topics/{topic}
    Headers: x-api-key: <key>. Body: {"message": "{\"v\":<version>}"} -
    the plan's 1KB law: send the POINTER, never the config body; Roblox
    fetches the full config via Module 4 on receipt. Guard in code:
    serialize, assert < 1024 bytes, throw if not (a failing guard is a
    bug in OUR code, not a retry case).
    Consumes: RestClient - the fluent chain:
        restClient.post().uri(...).header(...).body(...)
                   .retrieve().toBodilessEntity()
    non-2xx throws RestClientResponseException (that is the retry
    trigger). Built once via RestClient.builder() in a @Bean.

E3. TokenBucket                     (ours - rate limit respect)
    tryAcquire(): refill-on-read counter (capacity + refill rate from
    properties; Open Cloud's published limits set the numbers).
    Deliberately hand-rolled ~20 lines instead of a resilience library:
    one consumer, one bucket - a dependency would be bigger than the
    code.
    Consumes: nothing - ours (System.nanoTime()).

E4. EgressService                   (the internal API - what Module 4 calls)
    publishConfigActivated(placeId, version):
      up to 3 attempts, exponential backoff (1s, 2s, 4s - sleeps on the
      caller's virtual thread, cheap); each attempt gated by E3.
      On success: /topic/config delivery=PUSHED marker.
      On final failure: mark DEGRADED, publish /topic/alerts with the
      reason - and RETURN NORMALLY. No exception escapes to Module 4:
      activation already succeeded the moment the pointer moved.
    Consumes: @Service, RealtimePublisher.

----------------------------------------------------------------------
B. TEST PLAN (no real Roblox in tests, ever)
----------------------------------------------------------------------
Consumed test API: MockRestServiceServer bound to the RestClient.Builder
(@RestClientTest or manual bind) - scripts expected requests + canned
responses, verifies what was sent.
T1. Payload law: body sent is exactly {"v":N} pointer form, total < 1KB.
T2. Happy path: 200 -> one request, PUSHED marker published.
T3. Retry: two 500s then 200 -> exactly 3 requests, ends PUSHED.
T4. Degraded (acceptance): 401 x3 (bad key) -> DEGRADED alert published,
    method returns normally, no throw (poll path untouched - the
    Module 4 integration test T5 covers the pairing).
T5. Token bucket: burst of N+1 with capacity N -> last one waits or
    degrades per policy, never fires the HTTP call early.

----------------------------------------------------------------------
C. BUILD ORDER
----------------------------------------------------------------------
1. E1   2. E2 -> T1, T2   3. E3 -> T5   4. E4 -> T3, T4.
Real-universe verification (the phase's definition of done) happens
manually once, against a test universe, with the real key in env - never
in the automated suite.
