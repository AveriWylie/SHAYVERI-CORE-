SHAYVERI CORE - BUILD PLAN
===========================
Compute Orchestration & Rendering Engine. The Java backend for the SHAYVERI
Roblox pipeline. This document is the plan and the contract: the React
dashboard, node agents, and Roblox scripts are built against the surfaces
specced here, so they must not drift. Front-page/setup material lives in
README.md, not here.

----------------------------------------------------------------------
SYSTEM CONTEXT
----------------------------------------------------------------------

[ Roblox Game Servers ] --HTTP--> +----------------------+ --WebSocket--> [ React Dashboard ]
[ Lab Compute Nodes   ] --HTTP--> |    SHAYVERI CORE     |
                                  | (Java / Spring Boot) | --HTTPS--> [ Roblox Open Cloud API ]
                                  +----------+-----------+
                                             |
                                  [ MongoDB ]  [ Redis ]

CORE is the single source of truth. Roblox servers and lab nodes are
CLIENTS of CORE; they push data in and poll/receive config out. The
dashboard is a read/write control surface over WebSocket + REST. CORE is
the only component that talks to Roblox Open Cloud.

NON-GOALS (v1): no user accounts beyond API keys, no multi-tenant support,
no asset file storage inside CORE (files live on lab nodes / object
storage; CORE tracks metadata and status only).

----------------------------------------------------------------------
STACK DECISIONS (LOCKED)
----------------------------------------------------------------------
Language ......... Java 21 (LTS): virtual threads for cheap concurrency on polling/IO
Framework ........ Spring Boot 3.5.x: ecosystem, WebSocket/STOMP, Actuator, Data Mongo
                   (plan originally pinned 3.3.4; that version left Initializr's
                   supported range, 3.5.x satisfies the same requirement)
Build ............ Gradle (Kotlin DSL), committed wrapper
Primary DB ....... MongoDB: telemetry and job payloads are schema-fluid JSON
Hot state/queues . Redis: job queue, node heartbeats, live config cache, pub/sub
Realtime ......... WebSocket via STOMP (spring-boot-starter-websocket)
HTTP client ...... Spring RestClient (Open Cloud calls)
JSON ............. Jackson (bundled)
Local infra ...... Docker Compose (Mongo + Redis)

Security: see SECURITY ARCHITECTURE below.

----------------------------------------------------------------------
SECURITY ARCHITECTURE (designed to grow)
----------------------------------------------------------------------
Current model (v1): three static API-key roles - ROBLOX, NODE, DASH -
resolved from the X-Api-Key header via a OncePerRequestFilter. No
sessions, CSRF off, stateless. /actuator/health is the only public
endpoint. Keys come from environment variables, never committed.

Security lives in exactly TWO places, and nowhere else:
  1. The filter chain (authentication - "who are you"): an ordered stack
     of filters every request passes through before reaching any
     controller. Today: ApiKeyAuthFilter only. New concerns (rate
     limiting, request signing, IP allowlists) are ADDED AS NEW FILTERS
     in the stack; existing filters and controllers stay untouched.
  2. SecurityConfig (authorization - "what may you do"): endpoint-to-role
     rules declared centrally (e.g. only NODE may claim jobs). A protocol
     change is a one-line change in one file.

RULE (binding, same weight as dev rules 1-5): no module ever inspects
credentials itself. No controller or service reads the X-Api-Key header,
compares keys, or branches on identity. By the time a request reaches
module code, security has already decided. A credential check inside
ingress/, jobs/, etc. is an architecture violation.

Planned seam (build before Module 1 couples to the static impl):
  ApiKeyResolver (interface): resolve(rawKey) -> principal { role, label }
  StaticApiKeyResolver (v1 impl): the three env-var keys.
  The filter depends only on the interface. Future key storage changes
  (Mongo-backed keys, hashing, caching) swap the resolver; filter,
  config, and all modules stay unchanged. The label field future-proofs
  the Module 7 audit log ("node-7", not just "NODE").

Evolution roadmap - each future need lands in exactly one seam:
  per-client keys + revocation ....... new ApiKeyResolver impl
  key rotation via dashboard ......... new resolver impl + small DASH API
  rate limiting per key .............. new filter in the chain
  HMAC request signing / TLS ......... new filter / deployment concern
  OAuth dashboard login (v2 list) .... parallel filter path for DASH only
  tighter endpoint rules ............. SecurityConfig lines

PHASE 0 - SCAFFOLD (COMPLETE)
----------------------------------------------------------------------
Package layout (package-by-module, not by layer), under dev.shayveri.core:
config/ common/ ingress/ nodes/ jobs/ overrides/ egress/ realtime/
observability/.

Acceptance: ./gradlew bootRun starts against docker-compose Mongo/Redis;
GET /actuator/health returns UP; GET /api/ping returns
{"service":"shayveri-core","status":"ok"} and requires X-Api-Key.

----------------------------------------------------------------------
MODULE 1 - INGRESS (TELEMETRY INTAKE)
----------------------------------------------------------------------
Package: ingress. Callers: Roblox game servers (role ROBLOX).

Responsibility: accept telemetry POSTs from live Roblox servers, validate,
timestamp, persist, and fan out to the realtime hub. Must be cheap - this
is the highest-frequency path (every server instance, ~every 10s).

Endpoints:
  POST /api/telemetry
       body: { placeId, jobId, playerCount, serverFps, round, customMetrics{} }
       Returns 202 immediately; persistence and broadcast are async
       (virtual thread executor).
  POST /api/telemetry/events
       discrete events: player deaths (with position vectors for heatmaps),
       round transitions, perf spikes. Batched array accepted.

Owns (Mongo): telemetry_snapshots (TTL index, 7-day expiry - raw snapshots
are disposable); game_events (no TTL - feeds heatmaps and balance analysis).

Emits: /topic/telemetry/{placeId} on every accepted snapshot.

Acceptance: load-test 200 req/s sustained with p99 < 50ms; malformed
payloads return 400 with field errors, never 500.

(Construction detail: see module1_blueprint.txt in this folder.)

----------------------------------------------------------------------
MODULE 2 - NODES (LAB COMPUTE REGISTRY)
----------------------------------------------------------------------
Package: nodes. Callers: lab node agents (role NODE), dashboard (role DASH).

Responsibility: track the ~9 lab machines: registration, capabilities
(GPU, RAM, installed tooling e.g. Blender), heartbeat liveness, current
assignment.

Endpoints:
  POST /api/nodes/register
       node announces itself: { nodeId, hostname, capabilities{}, maxConcurrentJobs }
  POST /api/nodes/{id}/heartbeat
       every 15s; body includes current load. Stored in Redis with 45s
       expiry - a node with no heartbeat key is DOWN, no scheduler needed.
  GET  /api/nodes
       dashboard view: all nodes + live status merged from Redis.

Owns: Mongo nodes (durable registry); Redis node:hb:{id} (liveness),
node:load:{id}.

Emits: /topic/nodes on any status transition (UP->DOWN, idle->busy).

Acceptance: kill a node agent; dashboard shows DOWN within 60s. Node
re-registration is idempotent.

----------------------------------------------------------------------
MODULE 3 - JOBS (PIPELINE QUEUE & LIFECYCLE)
----------------------------------------------------------------------
Package: jobs. Callers: dashboard creates jobs, nodes claim/complete them.

Responsibility: the heart of the optimization pipeline. Jobs are units of
lab work: texture bake, mesh compression, light bake, PVS/pathfinding
pre-compute. CORE never executes work - it schedules, tracks, records.

Job model:
  Job { id, type: TEXTURE_BAKE|MESH_OPTIMIZE|LIGHT_BAKE|PATHFIND_PRECOMPUTE|CUSTOM,
        mapId, priority, payload{}, status: QUEUED|CLAIMED|RUNNING|DONE|FAILED,
        claimedBy, timings{}, resultRef (URI/path to artifact on node or
        object storage), resultMeta{} (e.g. tri counts before/after,
        texture sizes, bake duration) }

Endpoints:
  POST /api/jobs            (DASH)  enqueue; pushed to Redis list
                                    jobs:queue:{type} keyed by priority
  POST /api/jobs/claim      (NODE)  atomic claim (Redis LMOVE) filtered by
                                    node capability; long-poll up to 20s
  POST /api/jobs/{id}/progress (NODE) percent + log lines -> dashboard
  POST /api/jobs/{id}/complete (NODE) result metadata
  POST /api/jobs/{id}/fail     (NODE) error; FAILED jobs requeue up to
                                    maxRetries with backoff
  GET  /api/jobs?status=&mapId= (DASH) queue views

Owns: Mongo jobs (full history - the research dataset: every bake's
before/after metrics); Redis queues.

Emits: /topic/jobs (queue depth, transitions), /topic/jobs/{id} (progress).

Acceptance: two nodes claiming concurrently never receive the same job.
A node dying mid-job (heartbeat gone) triggers automatic requeue of its
CLAIMED/RUNNING jobs.

----------------------------------------------------------------------
MODULE 4 - OVERRIDES (GAME CONFIG CONTROL PLANE)
----------------------------------------------------------------------
Package: overrides. Callers: dashboard writes, Roblox reads.

Responsibility: versioned game-variable overrides: weapon stats, zombie
spawn rates, round pacing, graphics toggles. The "edit live from the
dashboard" feature.

Design: one active ConfigSet per place (or global). Every save creates a
new immutable version; the active pointer moves. Rollback = repoint. Full
audit trail of who changed what.

Endpoints:
  GET  /api/config/active?placeId=   (ROBLOX) the long-poll/fallback path:
       Roblox servers fetch on interval. Served from Redis cache,
       ETag-supported so unchanged polls are ~free.
  PUT  /api/config                   (DASH) save new version; validates
       against a registered schema per config namespace (weapons, spawns,
       graphics) so a typo'd JSON key can't brick a live game.
  POST /api/config/activate/{version} (DASH) repoint; triggers Module 5
       push + Redis cache refresh.
  GET  /api/config/history           (DASH)

Owns: Mongo config_versions; Redis config:active:{placeId}.

Emits: /topic/config on activation.

Acceptance: activating a version reaches a mock Roblox poller within one
poll interval AND via Open Cloud push within ~2s. Rollback restores prior
values exactly.

----------------------------------------------------------------------
MODULE 5 - EGRESS (ROBLOX OPEN CLOUD CLIENT)
----------------------------------------------------------------------
Package: egress. Callers: internal only (Modules 3/4 trigger it).

Responsibility: all outbound calls to Roblox. v1 = Open Cloud Messaging
Service: publish to a topic that live game servers subscribe to via
MessagingService, so config activations hit running servers instantly
instead of waiting for the next poll.

Internal API (Spring service, not HTTP):
  publishConfigActivated(placeId, version)
    -> POST apis.roblox.com/messaging-service/v1/universes/{universeId}/topics/{topic}
  Retry with exponential backoff (3 attempts); on final failure, mark
  delivery DEGRADED and broadcast a dashboard warning - the polling path
  in Module 4 is the guaranteed fallback, so push failure is never fatal.

Config: API key + universe ID from env (never committed). Respect Open
Cloud rate limits with a client-side token bucket.

Acceptance: with a bad API key, activation still succeeds (poll path) and
dashboard shows a degraded-push warning. Message payloads stay under
Roblox's 1KB messaging limit (send {v: version} pointer, not the config
body - Roblox fetches the full config via Module 4 on receipt).

----------------------------------------------------------------------
MODULE 6 - REALTIME (DASHBOARD WEBSOCKET HUB)
----------------------------------------------------------------------
Package: realtime. Callers: React dashboard (role DASH).

Responsibility: single STOMP broker relay pushing all live state to the
dashboard. Other modules call RealtimePublisher.publish(topic, payload) -
they never touch WebSocket APIs directly.

Topics:
  /topic/telemetry/{placeId}   live snapshots            (dashboard Tab 1)
  /topic/nodes                 node fleet status         (Tab 2)
  /topic/jobs, /topic/jobs/{id} queue + progress         (Tab 2)
  /topic/config                activation events         (Tab 3)
  /topic/alerts                degraded push, node down, job failure storms

Design notes: handshake auth via X-Api-Key in the CONNECT frame (or query
param) mapped to DASH. In-memory simple broker is fine for two users;
leave the config seam to swap to an external Redis-backed broker later.
Include a GET /api/snapshot REST endpoint returning current full state so
the dashboard can hydrate on connect before deltas stream in.

Acceptance: dashboard reconnect (kill wifi, restore) recovers full state
within 3s via snapshot + resubscribe, no duplicate or missed transitions
in the UI.

----------------------------------------------------------------------
MODULE 7 - OBSERVABILITY & AUDIT
----------------------------------------------------------------------
Package: observability.

Responsibility: know what CORE is doing without SSH-ing anywhere.
  - Micrometer metrics (via Actuator): telemetry ingest rate, queue depths
    (Redis gauge), Open Cloud call outcomes, WebSocket session count.
    Prometheus endpoint exposed; wire to Grafana later if wanted.
  - Audit log (Mongo audit): every DASH mutation - who (api key label),
    what, before/after diff. With two people editing live game values,
    "who changed the spawn rate" needs a definitive answer.
  - Structured JSON logging (logstash encoder) so lab-side log aggregation
    is trivial later.

Acceptance: /actuator/prometheus exposes custom shayveri_* metrics; every
override activation appears in the audit collection.

----------------------------------------------------------------------
DEVELOPMENT RULES
----------------------------------------------------------------------
1. One module per session/branch; each module lands as a coherent unit.
2. Integration tests over mocks: real Mongo and Redis via Testcontainers.
3. The public surface is a contract: endpoints, payloads, and topic names
   must match this document exactly - dashboard and node agents are built
   against it in parallel.
4. Secrets never committed: Open Cloud keys and production API keys come
   from environment variables.
5. Storage behind interfaces: each module defines a small store interface
   (JobStore, HeartbeatStore, ConfigStore...); Mongo/Redis implementations
   are adapters behind it. Business logic never touches a Mongo/Redis API
   directly. Keeps each storage slot independently swappable (see asdb
   under v2).
6. External APIs documented where used: anything a unit touches that we
   did not write (framework annotations, entry-point objects, test
   harnesses) is listed on that unit as a "Consumes" line in its module
   blueprint; the full catalog of consumed API surfaces lives in the
   Pre-Phase-1 summary PDF. Our own code needs no such line - it lives in
   its module, which IS its documentation of ownership.

----------------------------------------------------------------------
DEFERRED / V2 IDEAS (DO NOT BUILD YET)
----------------------------------------------------------------------
- Heatmap aggregation service (server-side binning of death positions).
- Object storage (MinIO) for artifact files instead of node-local paths.
- OAuth-backed dashboard login replacing static DASH key.
- Job DAGs (bake -> compress -> bundle as a chained pipeline instead of
  independent jobs).
- Per-map A/B config experiments with automatic telemetry comparison.
- asdb integration: swap one or more storage slots to asdb (custom Rust
  database) behind the existing store interfaces, starting with a low-risk
  slot (e.g. node heartbeats), once its server protocol and primitives
  (key expiry, atomic list ops) are confirmed.
