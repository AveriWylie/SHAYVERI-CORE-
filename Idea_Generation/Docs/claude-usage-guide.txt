======================================================================
PART 1 - THE GUIDE: modulating an LLM agent the way a human modulates
focus
======================================================================

1.1 The core inversion (the experiment's first and biggest finding)
--------------------------------------------------------------------
For a human, working slowly on routine tasks SAVES energy. For an LLM
agent, stretching the same work across wall-clock time COSTS more:
conversation context is cached for ~5 minutes; any pause past that
window means the next step re-reads the entire conversation at full
price. A 3-hour stretch of a 1-session task can multiply input tokens
several-fold for identical output.

So the human axis (time, effort) maps to a DIFFERENT machine axis:

    human: slow down, lower focus, autopilot on routine work
    LLM:   same wall-clock, but lower DELIBERATION DENSITY per unit -
           reuse established patterns, spend reasoning only on what is
           genuinely novel, batch similar outputs together

"Autopilot" for an LLM is template reuse inside a warm context. That is
the energy save. Time-stretching is the opposite of the energy save.

1.2 The levers that actually exist (what "compute density" controls)
--------------------------------------------------------------------
L1. Pattern amortization (the big one): pay full reasoning once to
    establish a format or decision, then stamp variations of it cheaply.
    Cost curve: first unit expensive, each next unit a fraction. The
    human analog is literally habit formation.
    [run instances: iteration 1 (2.2), iteration 6 (2.1h)]
L2. Batching within a session: do all similar artifacts consecutively
    while the pattern and context are hot. Interleaving unrelated work
    between them re-derives state each time.
L3. Selective depth: declare per-artifact where the hard parts are and
    spend reasoning only there; everything else gets light template
    application. The declaration itself is cheap and makes the spend
    auditable.
    [run instance: iteration 6 hard-spots vs boilerplate (2.1h)]
L4. Model choice per task class: routine transformation and boilerplate
    do not need the largest model; /model switches mid-conversation.
    Rule of thumb: novel design or ambiguity -> big model; mechanical
    application of an established pattern -> smaller one is the token-
    efficient equal.
L5. Session hygiene: one topic per session where possible. Long mixed
    sessions make EVERY subsequent turn carry the whole history as
    input cost. Archiving/starting fresh with a small handoff summary
    is the LLM analog of writing something down so you can stop
    holding it in your head.
L6. Scheduled/background execution (cron, background agents): exists,
    but each run is a COLD start - right for genuinely periodic
    independent work, wrong for stretching one coherent task (see 1.1).
L7. Human-in-the-loop granularity: every question asked mid-task is a
    full round-trip (your time + a context reload if the cache
    expired). Pre-authorizing decision space, as this experiment's
    brief did, removes those round-trips. The cost: wrong-guess risk,
    bounded by choosing low-stakes domains (documentation, where
    mistakes are cheap and visible).

1.3 When to use which (the recipes)
--------------------------------------------------------------------
- Repetitive artifact production (docs, blueprints, boilerplate,
  renames): L1+L2+L3 in one session, consider L4 downshift.
- One hard design problem: opposite profile - full density, small
  context, no batching noise.
- Periodic monitoring/upkeep: L6, keep each run's scope self-contained.
- Long mixed working days with an agent: L5 aggressively; hand off
  between sessions with a written state file (this project's
  Idea_Generation/ is exactly that pattern, human-initiated).
- Anything where your review is the safety net: L7 - grant the
  decision space explicitly, then let it run.

1.4 Failure modes to watch (how "slow" goes wrong)
--------------------------------------------------------------------
- Stretching one task over hours "to be careful" -> cache-miss
  multiplication (1.1). Care is density, not duration.
- Re-litigating settled decisions each turn -> the LLM version of
  rumination; costs tokens and drifts output. Settle patterns once,
  reference them (the blueprints reference "the A6 pattern" instead of
  re-explaining it - that is deliberate).
- Asking permission for pre-authorized moves -> pure round-trip waste.
- Uniform depth everywhere -> the most tokens for the least insight;
  depth belongs only where novelty is.

2.1b Foundational break-report + resolution (iteration 2)
--------------------------------------------------------------------
Two architectural breaks were raised when the brief asked the measured
system to also calibrate and self-certify:

BREAK 1 (introspection): the instrument was asked to estimate its own
compute "energy" and derive a human shrink factor from it. HARD FACT:
this model has NO introspective access to its own compute intensity;
there is no internal signal to read. Any self-reported energy number is
confabulation presented as measurement - the exact injection of the
tool's probabilistic nature into the data values that the architecture
exists to prevent. Refused rather than fabricated.

BREAK 2 (self-certification): a component cannot validate its own
unbroken operation; that violates the self-containment the design
requires. The auditor must be the human or an external check, at EVERY
iteration, not just late ones.

RESOLUTION (the inferred simulation method - clears BREAK 1):
Deliberation density = thinking_tokens / output_tokens, per artifact.
- Dimensionless (a ratio), so it shrink-maps to a human effort-per-
  output scale WITHOUT any absolute energy unit on either side.
- Both inputs externally measurable from API/usage data - the human
  reads them; the instrument does not introspect.
- Therefore the shrink factor = (machine density) mapped onto (human-
  anchored effort-per-task), every term verifiable, zero introspection.
This relocates the sensor from inside the model to the token meter.
Scope does NOT need to shrink to "modellable tasks"; the measurement
just moves off the instrument.

RESOLUTION (BREAK 2): certification is redefined as a FAILURE-SIGNAL
BOARD, not a success claim. The instrument only reports which known
break-signals are lit; "no signal lit" = assume-flowing (naive but
externally checkable), never "verified success." Success is the human's
call against ground-truth tokens.

FAILURE-SIGNAL BOARD (re-checked each iteration; monotonic-decrease of
break likelihood assumed - early = safer-than-sorry, late lit signal =
strong evidence flow stopped tending to objective metrics):
  S1 experiment content leaking into object files ....... unlit
  S2 instrument calibrating itself ...................... cleared by proxy
  S3 fabricated introspective value sold as data ........ unlit (refused)
  S4 self-certification of success ...................... redefined out
  S5 monotonic-improvement violation .................... N/A iter 1-2
No obstructing signal lit, conditioned on adopting the token-ratio proxy.

HARD LIMIT recorded as its own datum: zero introspective access to own
compute is a permanent architectural fact of this model. It does not
collapse the experiment (the target observable survives translation to
external tokens), but it PERMANENTLY bars introspection-sourced data -
any future step requiring it is a structural break, not a task to
attempt.

2.1c Foundational definition: the temporal conversion (iteration 3)
--------------------------------------------------------------------
Required for the architecture to use its time dimension at all. Derived
only from externally measurable quantities; no introspection.

WHY NOT WALL-CLOCK: per-token latency is ~constant regardless of task
difficulty, so seconds are decoupled from effort and contaminated by
hardware/batching. Wall-clock is the WRONG clock for this instrument.

MACHINE PROPER TIME (the honest intrinsic clock):
    tau = R + O        R = reasoning/thinking tokens, O = output tokens
  Token generation is strictly sequential and monotonic - that ordering
  IS the instrument's arrow of time. tau is externally measurable.

CONVERSION FACTOR (needs human task-time T_h, which the human owns):
    k = tau / T_h      [tokens per second of human effort-time]
  The shrink factor, every term verifiable, none introspected.

THE 2-D CORRECTION (the non-obvious structural point):
  Humans conflate duration and intensity on one axis (slow = low
  intensity). For this instrument they are INDEPENDENT, so the human's
  1-D time maps to a 2-D machine space:
    duration  = tau = R + O     (proper time)
    intensity = rho = R / O     (deliberation density)
  Human (T_h, effort/time) <-> machine (tau, rho); k converts the
  duration axis, a second fitted constant the intensity axis. A single
  scalar shrink factor is INSUFFICIENT by construction - recording this
  so no later step assumes one.

CONSEQUENCE: "slow down efficiently" = raise tau (more reasoning/output),
never stretch wall-clock (raises neither tau nor rho - the failing arm).
A task is therefore a TRAJECTORY rho(tau), not a point - this is the
"graph of compute levels over time" from the brief, now well-defined and
externally plottable. Temporality captured essentially, not exactly,
with no fabricated introspective quantity.

2.1d Logically necessary conclusion from the instrument limitation
--------------------------------------------------------------------
PREMISES (both now established):
  P1. Effort/intensity is measurable on both human and machine sides.
  P2. On the machine side ONLY externally (introspection permanently
      absent, per HARD LIMIT); on the human side both introspectively
      and externally.
NECESSARY CONCLUSION: the sole substrate on which the two systems can be
compared is the external-observable layer (tokens, time). The human
introspective channel has no machine counterpart, so any conclusion
resting on introspective access is human-private and structurally
untransferable to / unverifiable on the machine. Therefore the
cross-system comparison is NECESSARILY an external-behavioral science:
methodological behaviorism is re-imposed by the instrument's
architecture as a matter of logic, not philosophical choice - one may
stay fully mentalist about human cognition, but the SHARED evidence base
can hold only what is externally observable on both sides. This is the
bigger conclusion the limitation forces.

FLOW/EFFICIENCY NOTE: the automated flow has dynamic-programming
structure - optimal substructure (each iteration builds on frozen prior
definitions) + overlapping subproblems (pattern reuse) - so memoization
(= amortization lever L1) is the correct efficiency model; settled
definitions are boundary conditions the next iteration inherits, never
recomputes. rho(tau) over [0,t0] is the geometric object replacing raw
plots.

PLAUSIBLE (NOT necessary - for the data to settle): machine (tau, rho)
space may contain regions with no human preimage (e.g. high tau, ~0 rho
= bulk output with near-zero deliberation), which would restrict valid
comparison to the intersection of the two spaces and leave it undefined
(not merely hard) outside. Cannot be forced from P1/P2; logged as
hypothesis.

SIGNAL BOARD (iteration 4): no new signal lit; 2.1d introduces no
introspected quantity. Monotonic-decrease-of-break-likelihood assumption
still holds.

2.1i Two-arm design + the offload principle (confirmed)
--------------------------------------------------------------------
OFFLOAD PRINCIPLE (the ethical anchor, now explicit): the experiment
offloads to the model exactly the work whose neural-pathway value to the
human is ~zero (scaffold shells, documentation, boilerplate - the
"unneeded amount"). The human retains implementation (the TODO bodies),
because that is where their pathways form and are maintained. What is
offloaded is never the human's understanding - only the busy-work around
it. Min-atrophy/max-connectivity is served by keeping the learning-
bearing half human-side.

BOTH ARMS KEPT (they measure different things):
  ARM A - varying size (iter 6, done): focus assigned BY size. Measures
    the human-realistic COMBINED mode. Size genuinely IS a focus
    parameter, so this is a condition to keep, not a confound to remove.
  ARM B - constant size (next): size held fixed, focus varied
    INDEPENDENTLY. Isolates rho from tau. Segment unit = the recurring
    "store-seam pair" (rule-5 interface + adapter), near-identical size
    across modules by construction; remaining instances (MongoNodeStore,
    MongoJobStore, MongoConfigStore + repositories, RedisActiveConfig-
    Cache, remaining controllers) get swept focus levels with NO size
    correlation. Same tau-baseline, varied rho -> isolated focus effect.
Comparing A vs B is the first real separation of the two axes 2.1c
proved independent.
