MODULE 3 BLUEPRINT - JOBS (PIPELINE QUEUE & LIFECYCLE)
=======================================================
Phase 2 scope (with Module 2). The heart of the pipeline and the hardest
correctness territory in the system: two acceptance criteria here are
concurrency guarantees, not features. They are called out explicitly as
HARD SPOT 1 and HARD SPOT 2 - everything else is the established pattern.

----------------------------------------------------------------------
A. UNITS - package dev.shayveri.core.jobs
----------------------------------------------------------------------

J1. JobType / JobStatus             (enums)
    TEXTURE_BAKE | MESH_OPTIMIZE | LIGHT_BAKE | PATHFIND_PRECOMPUTE | CUSTOM
    QUEUED | CLAIMED | RUNNING | DONE | FAILED
    Consumes: nothing - ours. (Enums here, unlike A2's string type: job
    types are OUR vocabulary, a typo'd type must fail loudly at the edge.)

J2. JobCreateRequest                (record - DASH creates jobs)
    Fields: type (@NotNull JobType - Jackson maps the string, unknown
    value -> 400 via C2), mapId (@NotBlank), priority (@Min(0), default
    0), payload (Map, optional), maxRetries (@Min(0), default 3).
    Consumes: validation + Jackson enum binding.

J3. Job                             (Mongo document - the research dataset)
    Collection: jobs. Full history lives here forever; Redis only ever
    holds ids in flight.
    Fields: @Id String id (generated); JobType type; String mapId;
    int priority; Map payload; JobStatus status; String claimedBy;
    int attempts; int maxRetries;
    Timings as explicit Instants: createdAt, claimedAt, startedAt,
    finishedAt (the before/after research metrics live in resultMeta);
    String resultRef; Map resultMeta; String lastError.
    Consumes: @Document, @Id, @Indexed on status and mapId (the two
    query filters the plan's GET requires).

J4. JobStore                        (interface - rule 5)
    save(Job), findById, findByStatusAndMapId(optional filters),
    findByClaimedByAndStatusIn(nodeId, statuses)   <- exists FOR the
    orphan sweep; design the seam around the queries the system needs.
    Consumes: nothing - ours.

J5. MongoJobStore (+ JobRepository) - the A6/N5 pattern verbatim.
    Consumes: MongoRepository derived queries (findByStatus...,
    Spring parses the method names).

J6. QueueStore                      (interface - rule 5, hot path)
    enqueue(type, jobId, priority)
    claimOne(types, nodeId, timeout)  -> Optional<jobId>, blocks up to
                                         timeout (the 20s long-poll)
    release(nodeId, jobId)            -> back to its queue (requeue path)
    ack(nodeId, jobId)                -> done, drop from in-flight
    Consumes: nothing - ours.

J7. RedisQueueStore                 (adapter - HARD SPOT 1 lives here)
    Keys: jobs:queue:{type} (a list per type; priority handled by
    LPUSH/RPUSH choice: priority>0 -> RPUSH to a high list checked
    first, v1 keeps two bands, not a full priority queue - the plan
    says "keyed by priority", two bands satisfy 9 nodes),
    jobs:inflight:{nodeId} (what this node holds - the crash ledger).

    HARD SPOT 1 - the atomic claim. The acceptance criterion "two nodes
    claiming concurrently never receive the same job" is guaranteed by
    ONE property: Redis executes commands single-threaded, and
    BLMOVE source dest LEFT RIGHT timeout
    pops from the queue AND pushes onto the node's inflight list as one
    atomic command. There is no instant where the id is in neither or
    both lists, so no interleaving of two nodes can duplicate a claim.
    DO NOT decompose into read-then-move in Java: any check-then-act
    across two Redis calls reintroduces the race the acceptance test
    exists to catch.
    Consumes: StringRedisTemplate.opsForList() -
        leftPush(key, id); move(src, LEFT, dst, RIGHT) [LMOVE]; and the
        BLOCKING form via ops.move(..., Duration timeout) [BLMOVE] which
        parks the claim call server-side up to 20s - the long-poll with
        zero polling traffic. remove(key, 1, id) for ack/release.

J8. JobService                      (logic)
    create: build Job(QUEUED), J4.save, J6.enqueue, publish /topic/jobs
        (queue depth delta).
    claim(nodeId, capabilities): map capabilities -> claimable types;
        J6.claimOne over those types; on claim: status CLAIMED,
        claimedBy, claimedAt, attempts+1, save, publish.
    progress(id, pct, log): status RUNNING (first progress = startedAt),
        publish /topic/jobs/{id}.
    complete(id, resultRef, resultMeta): DONE, finishedAt, J6.ack, save,
        publish both topics.
    fail(id, error): attempts < maxRetries -> back to QUEUED with
        backoff delay (delay = base * 2^attempts; v1 implementation:
        virtual-thread sleep then enqueue - a crash during the sleep
        loses only the delay, the orphan sweep re-covers it),
        else FAILED terminal + /topic/alerts.
    Consumes: @Service, Executor (the AsyncConfig bean), RealtimePublisher.

J9. OrphanRequeueSweep              (HARD SPOT 2)
    The acceptance criterion: node dies mid-job -> its CLAIMED/RUNNING
    jobs requeue automatically. Design: @Scheduled sweep (30s) walks
    jobs:inflight:{nodeId} for every node the registry knows, asks
    Module 2's HeartbeatStore.isAlive(nodeId) - note: the INTERFACE
    from the nodes module, never RedisHeartbeatStore directly; this is
    rule 5 paying off across modules - and for dead nodes: release each
    inflight id back to its queue, flip Job to QUEUED (attempts intact),
    publish /topic/alerts.
    Idempotency guard: releasing an id that was already released (sweep
    raced a slow complete()) must be harmless - release() checks
    membership before moving; a completed job's ack already removed it.
    Consumes: @Scheduled + @EnableScheduling (shared config with N9);
    dev.shayveri.core.nodes.HeartbeatStore (ours, cross-module).

J10. JobController
    POST /api/jobs               (DASH)  202 + created id
    POST /api/jobs/claim         (NODE)  200 + job, or 204 after 20s empty
    POST /api/jobs/{id}/progress (NODE)
    POST /api/jobs/{id}/complete (NODE)
    POST /api/jobs/{id}/fail     (NODE)
    GET  /api/jobs?status=&mapId= (DASH)
    SecurityConfig: /api/jobs POST+GET -> DASH; /api/jobs/claim and
    /api/jobs/*/... -> NODE. (Order matters: claim's matcher before the
    /api/jobs/** DASH rule - first match wins.)
    Consumes: Spring Web set; claim's 20s block runs on a virtual
    thread (Tomcat is already virtual per application.yml, so a parked
    claim costs ~nothing - this is WHY the stack chose virtual threads).

----------------------------------------------------------------------
B. TEST PLAN
----------------------------------------------------------------------
T1. THE RACE TEST (acceptance): enqueue 1 job; two threads claim
    concurrently (CountDownLatch to release simultaneously); exactly one
    gets it, the other gets empty after timeout. Run 50 iterations.
T2. Claim respects capability: node without "blender" never receives
    TEXTURE_BAKE (given that mapping).
T3. Orphan requeue (acceptance): claim with node A, delete A's heartbeat
    key, run sweep -> job QUEUED again, inflight empty, alert published.
T4. Retry ceiling: fail a job maxRetries+1 times -> terminal FAILED,
    requeued exactly maxRetries times (attempts field proves it).
T5. Complete() racing the sweep: complete and sweep the same job ->
    job stays DONE, no duplicate in any queue (idempotency guard).
T6. Long-poll: claim on empty queue returns ~20s later empty; enqueue at
    t+5s -> claim returns immediately with the job.
T7. Roles per endpoint; GET filters by status and mapId.

5. J9 -> T3, T5   6. J10 + security -> full suite green.
