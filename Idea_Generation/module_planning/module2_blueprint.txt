MODULE 2 BLUEPRINT - NODES (LAB COMPUTE REGISTRY)
==================================================
Phase 2 scope (built alongside Module 3). Construction document, same
contract as Module 1's: units, responsibilities, surfaces, consumed APIs,
proof. Plan section: MODULE 2 in plan.txt.

Definition of done (phase-level, shared with Module 3): a dummy Python
node agent registers, claims a job, reports progress, completes. Module
2's own slice: kill the agent, dashboard shows DOWN within 60s;
re-registration is idempotent.

----------------------------------------------------------------------
A. UNITS - package dev.shayveri.core.nodes
----------------------------------------------------------------------

N1. NodeRegisterRequest             (record - incoming DTO)
    POST /api/nodes/register body.
    Fields: nodeId (@NotBlank), hostname (@NotBlank),
            capabilities (Map<String,Object>, optional -> Map.of() via
            compact constructor; e.g. {"gpu":"RTX 3080","blender":true}),
            maxConcurrentJobs (@NotNull @Min(1)).
    Consumes: Jakarta validation + Jackson (as Module 1 A1).

N2. HeartbeatRequest                (record - incoming DTO)
    POST /api/nodes/{id}/heartbeat body, every 15s from agents.
    Fields: currentLoad (@NotNull @Min(0) - running job count),
            runningJobIds (List<String>, optional).
    Consumes: same as N1.

N3. Node                            (Mongo document)
    Collection: nodes - the DURABLE registry (survives restarts; liveness
    deliberately does NOT live here, that is Redis's job).
    Fields: @Id String nodeId (natural key - the agent's own id, NOT a
    generated one: this is what makes re-registration idempotent, a
    second register with the same id is an upsert, not a duplicate);
    hostname; capabilities; maxConcurrentJobs; Instant registeredAt;
    Instant lastRegisteredAt.
    Consumes: @Document, @Id (as Module 1 A3; no TTL anywhere here).

N4. NodeStore                       (interface - rule 5 seam)
    save(Node), findAll(), findById(String).
    Consumes: nothing - ours.

N5. MongoNodeStore (+ NodeRepository)
    Adapter behind N4, exactly the Module 1 A6 pattern: package-private
    NodeRepository extends MongoRepository<Node, String>, adapter
    delegates.
    Consumes: MongoRepository (generated impl).

N6. HeartbeatStore                  (interface - rule 5 seam, hot state)
    recordHeartbeat(nodeId, load)   - write with 45s expiry
    isAlive(nodeId)                 - key exists?
    aliveNodeIds()                  - all currently-live ids
    loadOf(nodeId)                  - current load or empty
    This is the asdb-candidate slot named in the plan's v2 list.
    Consumes: nothing - ours.

N7. RedisHeartbeatStore             (adapter behind N6)
    THE DESIGN (from the plan): a node with no heartbeat key IS down -
    the database's key expiry is the failure detector, no scheduler
    tracks timeouts.
    Keys: node:hb:{id} (value = timestamp, TTL 45s),
          node:load:{id} (value = load, TTL 45s).
    Consumes: StringRedisTemplate -
        opsForValue().set(key, value, Duration.ofSeconds(45)) - write+TTL
        in one call; hasKey(key); keys("node:hb:*") for aliveNodeIds()
        (fine at 9 nodes; SCAN if the fleet ever grows).

N8. NodeService                     (logic)
    register(req): upsert into N4, stamp lastRegisteredAt; returns same
        result for first and repeat registration (idempotency).
    heartbeat(id, req): N6.recordHeartbeat; unknown id -> reject (agents
        must register first).
    listWithStatus(): N4.findAll() merged with N6 liveness -> per node
        {node, status: UP|DOWN, load}. Status is COMPUTED AT READ TIME
        from key existence, never stored.
    Consumes: @Service.

N9. NodeStatusSweep                 (the one novel unit in this module)
    Problem it solves: Redis key expiry is silent - nothing fires when a
    node dies, but the plan requires a /topic/nodes broadcast on
    UP->DOWN. Solution: a 15s scheduled sweep compares current liveness
    against the previous sweep's snapshot (kept in memory - loss on
    restart is harmless, next sweep rebuilds it) and publishes only the
    TRANSITIONS. Worst-case detection: 45s TTL + 15s sweep = within the
    plan's 60s acceptance bound - the numbers are chosen by that bound.
    Consumes: @Scheduled(fixedDelay = 15000) on the method +
    @EnableScheduling once on a config class (without it @Scheduled is
    silently inert - same trap as auto-index-creation);
    RealtimePublisher (B1, ours) for /topic/nodes.

N10. NodeController                 (HTTP edge)
    POST /api/nodes/register   (NODE)
    POST /api/nodes/{id}/heartbeat (NODE)
    GET  /api/nodes            (DASH)
    202/200s, thin delegation, zero identity checks (security
    architecture). SecurityConfig lines:
      .requestMatchers("/api/nodes/register", "/api/nodes/*/heartbeat").hasRole("NODE")
      .requestMatchers("/api/nodes").hasRole("DASH")
    Consumes: Spring Web set (as Module 1 A8).

B. TEST PLAN (Testcontainers Mongo + Redis; Redis container joins here:
   org.testcontainers redis via GenericContainer("redis:7-alpine") +
   @ServiceConnection, or spring-boot-testcontainers' RedisContainer)
----------------------------------------------------------------------
T1. Register twice with same nodeId -> one document, lastRegisteredAt
    updated, both calls succeed identically (idempotency).
T2. Heartbeat -> node:hb:{id} exists with TTL between 0 and 45s
    (redisTemplate.getExpire(key)).
T3. Liveness: heartbeat then wait past TTL (test with a short override,
    e.g. 2s TTL injected via property) -> isAlive flips true->false.
T4. Sweep transition: force UP->DOWN, assert exactly one /topic/nodes
    message (subscribe a STOMP test client, Module 1 D4 pattern).
T5. Roles: DASH key on register -> 403; NODE key on GET /api/nodes -> 403.
T6. GET /api/nodes merges: registered node without heartbeat -> DOWN,
    with heartbeat -> UP with load.
