MODULE 1 BLUEPRINT - INGRESS (+ MINIMAL REALTIME)
==================================================
Phase 1 scope. This is a construction document, not code: every unit that
will exist, what it is responsible for, its public surface, and how it will
be proven. Implementation happens against this document; the plan
(README.md / plan.txt) stays the outer contract.

Definition of done for the whole phase: a fake Luau script (a curl loop)
posts telemetry, and it is visible live on a raw WebSocket client.

----------------------------------------------------------------------
A. UNITS TO CREATE - package dev.shayveri.core.ingress
----------------------------------------------------------------------

A1. TelemetrySnapshotRequest        (record - incoming DTO)
    Responsibility: the exact shape of POST /api/telemetry's JSON body.
    Fields: placeId (required), jobId (required), playerCount (>= 0),
            serverFps (> 0), round (free string), customMetrics (map,
            optional, defaults empty).
    Validation lives here as annotations (@NotNull, @Min...), nowhere else.
    Done when: a malformed body is rejected before any logic runs.

A2. GameEventRequest                (record - incoming DTO)
    Responsibility: one discrete event in POST /api/telemetry/events.
    Fields: type (enum-like string: PLAYER_DEATH, ROUND_START, ROUND_END,
            PERF_SPIKE, CUSTOM), placeId, jobId, occurredAt (client
            timestamp), position (optional {x,y,z} for heatmaps),
            data (map, optional).
    The endpoint accepts a JSON ARRAY of these (batching).

A3. TelemetrySnapshot               (Mongo document)
    Responsibility: persisted form of A1 + server-side receivedAt.
    Collection: telemetry_snapshots. TTL index on receivedAt, 7 days.
    Note: TTL index must be declared programmatically/config so it exists
    in any fresh database, not hand-created.

A4. GameEvent                       (Mongo document)
    Responsibility: persisted form of A2 + receivedAt. Research data.
    Collection: game_events. NO TTL - kept forever for heatmaps/balance.

A5. TelemetryStore                  (interface - the storage seam)
    Responsibility: the ONLY doorway to persistence for this module
    (development rule 5; the future asdb swap point).
    Surface: saveSnapshot(TelemetrySnapshot),
             saveEvents(List<GameEvent>).
    Nothing else in the module may import a Mongo API.

A6. MongoTelemetryStore             (class - adapter implementing A5)
    Responsibility: Spring Data Mongo implementation of TelemetryStore.
    The only file in the module that knows Mongo exists.

A7. TelemetryService                (class - the logic)
    Responsibility: orchestrate accept -> stamp receivedAt -> persist via
    A5 -> publish to realtime via B1. Runs async on the virtual-thread
    executor so the controller can return 202 immediately.
    Surface: accept(TelemetrySnapshotRequest),
             acceptEvents(List<GameEventRequest>).

A8. TelemetryController             (class - the HTTP edge)
    Responsibility: two endpoints, role ROBLOX only, both return 202
    Accepted with no body, delegating instantly to A7.
      POST /api/telemetry          <- A1 body
      POST /api/telemetry/events   <- array of A2
    No logic here beyond delegation; thin by design.

----------------------------------------------------------------------
B. UNITS TO CREATE - package dev.shayveri.core.realtime (minimal slice)
----------------------------------------------------------------------

B1. RealtimePublisher               (interface)
    Responsibility: the one façade every module uses to broadcast; no
    module ever touches WebSocket APIs directly (plan, Module 6).
    Surface: publish(String topic, Object payload).

B2. StompRealtimePublisher          (class - adapter implementing B1)
    Responsibility: delegate to Spring's SimpMessagingTemplate.

B3. WebSocketConfig                 (class - in config package)
    Responsibility: enable the STOMP simple broker; endpoint /ws;
    broker prefix /topic. Handshake auth (DASH key in CONNECT) is
    Phase-1-minimal: permitted for now, hardened when Module 6 is built
    properly. Leave the seam obvious.

Topic used this phase: /topic/telemetry/{placeId} on every accepted
snapshot (payload = the persisted snapshot).

----------------------------------------------------------------------
C. UNITS TO CREATE - package dev.shayveri.core.common (shared, built here
   because Module 1 is the first to need them)
----------------------------------------------------------------------

C1. ApiError                        (record - uniform error body)
    Fields: status, message, fieldErrors (map field -> problem), timestamp.

C2. GlobalExceptionHandler          (class - @RestControllerAdvice)
    Responsibility: convert validation failures and bad JSON into 400 +
    C1 with per-field detail; convert anything unexpected into a clean
    500 WITHOUT a stack trace in the response. Plan's acceptance:
    "malformed payloads return 400 with field errors, never 500."

----------------------------------------------------------------------
D. TEST PLAN (Testcontainers: real Mongo + Redis, no mocks - dev rule 2)
----------------------------------------------------------------------

D1. Validation tests (web layer):
    - missing placeId -> 400, fieldErrors names placeId, never 500
    - negative playerCount -> 400
    - garbage non-JSON body -> 400 with clean ApiError
    - valid body -> 202 and empty response

D2. Security tests:
    - no X-Api-Key -> rejected
    - DASH or NODE key on /api/telemetry -> rejected (ROBLOX only)
    - ROBLOX key -> 202

D3. Persistence tests (Testcontainers Mongo):
    - accepted snapshot appears in telemetry_snapshots with receivedAt set
    - TTL index exists on telemetry_snapshots with 7-day expiry
    - batch of 3 events -> 3 documents in game_events, no TTL index

D4. Realtime test:
    - subscribe a test STOMP client to /topic/telemetry/123, POST a
      snapshot with placeId 123, assert the payload arrives
    - snapshot for placeId 456 does NOT arrive on topic .../123

D5. Async behavior test:
    - controller returns 202 even if Mongo is slow (persistence is not on
      the request thread)

D6. Manual end-to-end check (the phase's definition of done):
    - docker compose up, bootRun, run scripts/fake-roblox.sh (curl loop
      posting snapshots every 2s), watch a raw WebSocket/STOMP client
      receive them live. Script to be written alongside the module.

Deferred out of this phase deliberately: the 200 req/s p99 < 50ms load
test (plan lists it under Phase 4 hardening); Redis is untouched by
Module 1 (first used by Module 2).

----------------------------------------------------------------------
E. BUILD ORDER WITHIN THE MODULE (each step compiles + its tests pass
   before the next starts)
----------------------------------------------------------------------
 1. C1 + C2 (error contract first - everything else returns it)
 2. A1 + A2 (request shapes)     -> D1 partially runnable
 3. A3 + A4 (documents)
 4. A5 + A6 (store seam + Mongo adapter)   -> D3
 5. B1 + B2 + B3 (publisher + websocket config)
 6. A7 (service wiring store + publisher, async)   -> D5
 7. A8 (controller)   -> D1, D2 fully green
 8. D4 realtime test, then D6 manual end-to-end
